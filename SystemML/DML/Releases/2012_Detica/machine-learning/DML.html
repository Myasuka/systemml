    Comments

    DML supports two forms of commenting: line and block comments. A line comment is indicated using a hash (#), and everything to the right of the hash is commented out. A block comment is indicated using “/*” to start the comment block and “*/” to end it.

    # this is an example of a line comment

    /* this is an example of a

      multi-line block comment

    */

    Data Types and Variables

    DML supports 3 data types: matrices, scalars, objects, and 5 value types: double, integer, string, boolean, and object. Matrices only support the double value type (i.e., the cells in a matrix can only be doubles). The data type object refers to any object that is neither a matrix or a scalar (e.g., a binary representation of a decision tree). The data type object can only be used with the value type object. The value type object can only be used with the data type object.

    Each matrix has a specified number of rows and columns. A 1 x 1 matrix is not equivalent to a scalar double. DML only supports two dimensional matrices. The first index for both row and columns in a matrix is 1. For example, a matrix with 10 rows and 10 columns would have rows numbered 1 to 10, and columns numbered 1 to 10.

    The elements of the matrix can be accessed by indexing. The indexing must specify both the row and column values, separated by a comma. The row and column values could either be integer values, other scalar integer variables in the DML program, or an expression that evaluates to an integer value. To select the entire row or column of a matrix, a colon is used for the index of the appropriate dimension.

    Examples:

    X[1,4] # access cell in row 1, column 4 of matrix X

    X[i,j] # access cell in row i, column j of X.

    X[1,:] # access the 1st row of X

    X[:,2] # access the 2nd column of X

    X[: , :] # access all rows and columns of X

    DML also supports range indexing to access a contiguous block of rows and columns in the matrix. The grammer for row-based indexing is below. The constraint is that lower-row < upper-row, and lower-column < upper-column.

    [Marix name][lower-row : upper-row],[lower-column : upper-column]

    Examples:

    X[1:4, 1:4] # access the 4 x 4 submatix comprising columns 1 – 4 of rows 1 – 4 of X

    X[1:4, : ] # select the first 4 rows of X

    Statements

    A DML program consists of a sequence of statements, with the default computation semantics being sequential evaluation of the individual statements. The types of statements supported currently in DML are file input/output (Read, Write), assignment, control structures (While, If, For), and user-defined function statements (UDF declaration, UDF function call).

    Similar to R, the use of a semi-colon at the end of a statement is optional.

    File format and MTD files

    The Read/Write statements support reading / writing all data types (matrices, scalars, objects) from/to files. The files are not managed in memory but kept in HDFS and processed by Hadoop. Associated with each data file is a optional metadata file (MTD) which stores metadata information about the contents of the data file. For data files generated by SystemML, a MTD file will automatically be generated. The name of the MTD file associated with the data file <filename> must be <filename>.mtd. For example, if SystemML writes out a data file X.mtx to HDFS, the name of the data file will be X.mtx and the generated MTD file will be X.mtx.mtd. A user can specify an MTD file for their own data as well.

    SystemML supports a sparse text format on disk for matrices. The cell values are stored in (row column value)-format with the row and column indices being 1-based. The binary format can only be read and written by SystemML. DML scripts with read and write statements can be executed to perform format conversions.

    Examples:

    3 4 10.0
    5 6 7.0

Read Statement

    The read statement is used to read in all data types (matrix, scalar, object) from data files for use in SystemML. The syntax of a read statement is as follows:

    read(“inputfile”, [list of parameters])

    where inputfile is the path to the data file in HDFS. The list of parameters are the same as given in Table 1. For the read statement, the user has the option of specifying each parameter value either in the MTD file, the read statement, or in both locations. However, parameter values specified in both the read statement and MTD file must have the same value. Also, if a scalar value is being read, then format cannot be specified. The read statement in SystemML is parameterized as follows during compilation.

    1. Default values are assigned to parameters with default values.

    2. Parameters provided in the read statement either fill in values or override defaults. Cann

    3. SystemML will look for the MTD file at compile time in the specified location (same path as data file, filename for MTD file is same name as data file with ”.mtd” appended).

    4. If all non-optional parameters are specified or conflicting values are detected, then an exception is thrown.

Examples:

# read a matrix data file from HDFS with path “in/v”. The default data_type value of matrix and default value_type of double are used.

V = read ("in/v", rows=10, cols=8, format="text");

    # read a matrix data file from HDFS with an MTD file available (MTD file path: in/data.mtd)

    V = read(“in/data”);

    # read a scalar integer value from HDFS file “in/scalar_i”

    V = read(“in/scalar_i”,data_type=”scalar”, value_type=”int”);

      Assignment Statement

    An assignment statement consists of an expression, the result of which is assigned to a variable. The variable gets the appropriate data type (matrix or scalar) and value type (double, int, string, boolean) depending on the type of the variable output by the expression.

    Examples:

    # max_iteration is of type integer
    max_iteration = 3;

    # V has data type matrix and value type double.

    V = W %*% H;

    The operators allowed in expressions in an assignment statement are described below under Expressions.

      Control Statements

DML currently supports 3 control statements: while, if, and for.

While Statement

    The syntax for a while statement is as follows.

    while (predicate){

      statement1

      statement2

      ...

    }

    The statements in the while statement body are evaluated repeatedly until the predicate evaluates to true. The while statement body must be surrounded by braces, even if the body only has a single statement.

    The predicate in the while statement consist of operations on scalar variables and literals. The body of a while statement may contain any sequence of Read/Write, Assignment, Control structures (while, if, for), and UDF function statements. Thus, DML supports nested control structures.


    Example:

      while( (i < 20) & (!converge) ) {

        H = H * (t(W) %*% V) / ( t(W) %*% W %*% H);
        W = W * (V %*% t(H) / (W %*% H %*% t(H));
        i = i + 1;

      }




If Statement

    The syntax for an if statement is as follows.

if (predicate){

statement1

statement2

…...

[else {

statement1

statement2

…....

}]?

Conceptually, the If statement has two bodies: the if body (evaluated if the predicate evaluates to true) and the optional else body (evaluated otherwise). Both the statements in the if body and else body must be surrounded by braces, even if the body only has a single statement. The if body and else body may contain any sequence of Read/Write, Assignment, Control structures (while, if, else), and UDF function statements. Thus, DML supports nested control structures, such as in the second example below.

    Examples:

    # example of if statement

if( i < 20 ) {

converge = false;

}

else {

converge = true;

}

# example of nested control structures

while( !converge ) {

H = H * (t(W) %*% V) / ( t(W) %*% W %*% H);
W = W * (V %*% t(H) / (W %*% H %*% t(H));
i = i + 1;

zerror = z - H %*% W;

if (zerror < maxError){

converge = true;

}

else {

converge = false;

}

      }


For Statement

    The syntax for a for statement is as follows.

    for (var in [lower]:[upper]){

      statement1

      statement2

      …...

}

var is an integer scalar variable. lower and upper are integers, and [lower]:[upper] define a sequence of numbers{ lower, lower + 1, lower + 2, … , upper – 1, }. For each element in the sequence, var is assigned the value, and statements in the for loop body are executed.

The for loop body may contain any sequence of Read/Write, Assignment, Control structures (while, if, else), and UDF function statements. Thus, DML supports nested control structures. The statements in the for statement body must be surrounded by braces, even if the body only has a single statement.

    Example:

      # example for statement

      A = 5;

      for( i in 1:20) {

        A = A + 1;

      }


    User-defined Function (UDF) Statements

There are two DML statements that address declaring and calling user-defined functions. A UDF function declaration statement provides the function signature, which defines the formal parameters used to call the function and return values for the function. The function definition specifies the function implementation, and can either be native DML code or external packages / libraries. If the UDF is implemented in DML, then the DML for both UDF declaration and definition occur together. A UDF function call statement is used to invoke the function within a DML script.

Function declaration statement

The syntax for the UDF function declaration for DML-bodied functions is given as follows. The function definition is regular DML code, which is stored as a list of statements in the function body. The explination of the parameters is given in Table 3. The statements that can be placed inside a UDF defintion include control structures, assigment statements, buildin functions, and UDF function calls. However, a UDF function declaration statement cannot be placed inside another UDF function declaration statement. The variables specified in the return clause will be returned, and no explicit return statement within the function body is required.

functionName = function([ <DataType>? <ValueType> <var>, ]* ) return ([ <DataType>? <ValueType> <var>,]*)

{

# function body definition in DML

statement1

statement2

....

}

The syntax for the UDF function declaration for functions defined in external pacakages / libraries is given as follows. The parameters are explained in Table 3. The main difference is that a user must specify the appropriate collection of userParam=value pairs (i.e., parameter name, followed by a '=', followed by value) for the given external pacakage.

functionName = externalFunction([<DataType>? <ValueType> <var>, ]* ) return ([ <DataType>? <ValueType> <var>,]*) implemented in ([userParam=value]*)

UDF Function Call Statement

A UDF is invoked within a DML script using a Function Call Statement. The function call statement specifies the function identifier, variable identifiers for calling parameters, and the variables to be populated by the returned values from the function. DML supports functions which return multiple values. The syntax for function calls is as follows.

returnVal = functionName(param1, param2, ….)

[returnVal1, returnVal2, ...] = functionName(param1, param2, ….)

Example:

# DML script with a function call

B = zeros(10,10);

C = zeros(100,100);

D = addEach(1, C);

index = 0

while (index < 5) {

[minD, maxD] = minMax(D)

index = index + 1

}

    Expressions

    Expressions in DML apply an operation, such as matrix multiplication, over one or more input operands. The operations in DML are divided into operators such as matrix multiplication and builtin functions such as diag and transpose.

    Operators

    In addition to linear algebra operations such as matrix multiplication, matrices can also be used in arithmetic, relational and boolean expressions. In these cases, the operations are performed element by element. For instance,

    A = B * C;

    is evaluated by multiplying each cell in B with the corresponding cell in C. Note that the dimensions of the input matrices need to match based on the operator semantics; otherwise an exception will be raised at compile time. When one of the operands is a matrix and the other operand is a scalar, the operation is performed elementwise on the matrix using the scalar operand. For example

     A = B * 20;

     is evaluated by multiplying every cell in B by the scalar value 20.

      Builtin Functions

      The builtin functions supported in DML can be broadly classified as follows.

    • Linear algebra : Functions applying linear algebra operations on matrices such as transpose, diag and trace.

    • Matrix Aggregation : Functions performing aggregation operations on matrices, such as sum, prod, avg, min and max.

    • Matrix Partial Aggregation : Functions aggregating matrices along a single dimension such as rowSums and colSums.

    • Matrix Dimensionality : Functions to compute dimensionality of a matrix, such as length, nRow and nCol.

    • Comparison : Functions comparing two operands; for matrices the functions supported are pmin and pmax and for scalars the corresponding functions are min and max respecively.

    • Mathematical : Mathematical and trigonometric functions, such as sqrt, log, abs, sin, cos and tan. These functions can be applied on matrices or scalars. 

    • Cast : Functions to cast objects from one type to another. E.g., castAsScalar.

    The table below lists the functions currently supported in DML along with a description of the input and output types for each of them. For example, consider the following expressions.

    s = sum(A);
    B = rowSums(A);
    C = colSums(A);
    D = rowSums(C);
    diff = s - castAsScalar(D);


    The builtin function sum operates on a matrix (say A of dimensionality m x n) and returns a scalar value corresponding to the sum of all values in the matrix. The builtin functions rowSums and colSums, on the other hand, aggregate values on a per-row and per-column basis respectively. They output matrices of dimensionality mx1 and 1xn respectively. Therefore, B is a mx1 matrix and C is a 1xn matrix. Applying rowSums on matrix C, we obtain matrix D as a 1x1 matrix. In DML, a 1x1 matrix is different from a scalar; to treat D as a scalar, an explicit castAsScalar operation is invoked in the final statement. The difference between s and castAsScalar(D) should be 0.

Modules

DML modules allow users to package related DML functions so that they are easily re-usable by others. A DML module is implemented in a single DML file which contains a collection of UDF Function Declaration statements. The features that DML's modules enable include:


  • Namespaces: functions defined in a module are placed in their own namespace to avoid name clashes

  • Packages: organize modules using directories.

Search Path

The module search path is a string containing file system paths that are delimited by ":". The way to specify the search path to the DML interpreter is as follows.


dml-path "projects/DML/myModules:scratch/myOtherModules"


The module search path is used to look up module and package files during import.



Import

Several options are available when importing a module to control how the module's variables interact with the current namespace. The current namespace is defined to be the namespace of the script that issuing the import call (e.g., a script or the interpreter).


# will import functions defined in simple1.dml into new namespace

import simple1;


# reference functions as

simple1::foo()


# promote only function foo into current namespace

import simple1 (foo);


# promotes all functions declared in simple1 into current namespace

import simple1 (*);


# alias for simple1

import simple1 as x;

x::foo();


In addition, the import can specify a module that is contained within packages.

Suppose that simple1 is nested in the package test, e.g., /home/joeuser/modules/test/simple1. Then the import would look like this:


import test::simple1;



Packages

DML packages correspond to directories (similar to package design in Python).

A package can include other packages and modules.

For example, consider the following directory layout:

/home/joeuser/modules/p1/simple1.dml

/home/joeuser/modules/p1/p2/simple2.dml

/home/joeuser/modules/p3.dml

/home/joeuser/modules/p3/simple3.dml


Four modules are defined in three packages (i.e., each of the 4 DML files defines a module) . The p1 package includes a module (p1::simple1) and a package that includes a module (p1::p2::simple2).

dml-path '<pathset>'


import <module-name> [ ( [ <var-list> | *] ) ] [ as <alias> ]
